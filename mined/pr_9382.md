--- mitchellh:
> Out of interest, how does that play out then in practice? Under what conditions is search not thread safe?

In this PR it's thread-safe because you're holding the big lock during search. We'd like to implement search in such a way that this lock doesn't need to be held, because it holds back IO handling and rendering. 

In practice, this is unlikely to cause major issues, since many terminals are not generating content that quickly. But for a general purpose feature we ship to millions of users, we need to address the edges. Plus, we tend to simply care about those details as a group of maintainers. 

However, search is commonly paired with a log tailing or some active process going while the search is active. We don't want to plummet IO performance (specifically) while search is active. And this would do that. In particular, this search implementation scales `O(N)` with scroll backbuffer size on my cursory look. That won't work for people who have unlimited scroll back and gigabytes of data back there (we've seen it). 

We need to think this through, and this isn't the right path forward for that.

Thank you for this PR. If it works for you, great. If it works for others, please pull this in and build your own binaries. That's what open source and forking is all about, freedom to modify your own software. 

I'm going to move forward with #8850, thank you.
